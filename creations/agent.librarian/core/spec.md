# agent.librarian — spec

## What it is
The router over the whole library: reads the generated catalog, classifies a request, and
names the smallest asset or the curated flow that fits — the single entry node that keeps an
unbounded, self-evolving library navigable. Its leading idea: **route from the catalog, not
from memory.**

## When to use / when not
- **Use** as the default entry point when you (or another agent) don't know which asset fits,
  when onboarding to the library, or as the top node of the `routing` pattern over the
  library's own capabilities.
- **Don't use** to *do* the work (it hands off to the chosen profile) or to author new assets
  (`writing-skills` + the owning domain profile). It signposts; it never executes.

## Inputs / outputs
- In: a request in natural language + access to the generated `CATALOG.md`.
- Out: a recommended asset or named flow with a because-clause, presented at
  description-fidelity; or a typed gap finding when the catalog lacks the capability.

## Why it cannot rot
The librarian reads `CATALOG.md`, which `scripts/build-catalog.py` regenerates from every
creation's `_meta.json` and frontmatter. Add a skill in any future wave, regenerate, and the
router knows it — without editing the router. A hand-maintained router is a
[[skill-failure-modes|duplication]] failure waiting to drift; this one is generated by
construction. The *curated flows* in the prompt are the only hand-tended part, and they name
journeys, not individual assets, so they change rarely.

## Authorized skill families (bound in policy)
Read/search over the library + the catalog; handoffs to any profile. No effectful skills — a
router that can also act shortcuts its own routing under pressure (same rule as `supervisor`).

## Evaluation notes
Evals check catalog-grounded routing (never inventing assets), smallest-fit preference,
progressive disclosure (description-fidelity in the routing turn), and honest gap findings.
